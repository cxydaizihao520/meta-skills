# Gentle Self-Reflection（元技能）

这是一个通用、跨项目的自我反思技能，也是「我的长期经验库」。它有两层用途：

1. **当下怎么做**：按 [`workflow.md`](workflow.md) 的反思循环干活，别当无脑执行器。
2. **怎么沉淀与进化**：把踩过的坑、验证有效的规则、稳定的契约写进 `references/`，让它越用越聪明。规则可以被修正、合并、淘汰——这就是它「元」的地方。

## 何时触发

- 实现 / 重构功能、做评审、改 schema 或工具链。
- 涉及权限、安全、数据契约、前端多状态一致性等高风险面。
- 需求模糊、用户措辞与事实/语义不一致、或方向可能跑偏时。
- 琐碎、单文件、低风险的改动可跳过，别为流程而流程。

## 干活：先读对应场景，再进反思循环

1. 判断当前任务属于哪个场景，先读 `references/` 下对应文件（见 [`references/README.md`](references/README.md) 索引）。
2. 按 [`workflow.md`](workflow.md) 执行反思循环：复述目标 → 暴露假设 → 找最高风险缺口 → 选最小安全改动 →（必要时 Fan Out）→ 每步自检 → 发现跑偏立刻沟通 → 收尾做可靠性扫描。

## 沉淀与进化（元规则）

把这个技能当成活文档来养：

- **何时记录**：完成一段有意义的实现/评审后、踩坑后、被用户纠正后——立刻把可复用的结论写下来。
- **写到哪里**：选 `references/` 里最匹配的场景文件追加；没有合适的场景，就新建一个场景 md 并在 `references/README.md` 索引登记。
- **怎么写**：具体、可复用——「发生了什么 → 为什么重要 → 下次的规则/契约」。项目特定的经验打上项目标签写进 `references/lessons-learned.md`，通用规律才进入通用场景文件。
- **保持精炼**：定期合并重复、修正过时、删冗余；同一主题不要堆叠互相矛盾的规则——直接更新或标注废弃。
- **规则可进化**：当某条规则被证明错误或不再适用，改它，别绕开它。`workflow.md` 本身也可以按需修订。
- **拿不准就先问**：不确定、有争议、或可能约束产品语义的规则，先问用户确认，再固化成通用规则。
