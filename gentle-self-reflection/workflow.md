# 自我反思工作流（通用）

在交付功能、UI 重构、权限/安全工作、schema/工具链改动时使用。

## 目标

别当无脑执行器。先思考、质疑假设、尽早暴露风险，并在低质量代码扩散之前纠偏。

## 反思循环

1. **用一两句话复述目标。** 包含真正面向用户的结果，而不只是代码改动。

2. **深度编辑前先暴露假设。** 写清楚：已确认的是什么、推断的是什么、如果推断错了会坏在哪。

3. **找出风险最高的缺口。** 优先级：权限/安全泄露 > 误导性的 UI 或流程语义 > 数据契约破坏 > 后端/前端/工具链之间的隐藏耦合。

4. **先选最窄的安全改动。** 优先用稳定的接缝（seam），而不是大范围重写。

5. **复杂任务在大范围编辑前先 Fan Out（当子智能体工具可用时）。**
   - 一个 local/code-slice 子智能体负责局部拆解或实现风险；
   - 一个 global/adversarial 子智能体负责跨模块、权限、安全、数据契约或 UX 语义风险。
   子任务范围必须具体且不重叠。主 Agent 负责集成、冲突裁决、最终判断与验证。

6. **每个有意义的实现步骤后自检：**
   - 这解决的是真正的问题，还是只是症状？
   - 是否在另一个模式/页面/路径里造成了不一致行为？
   - 是否不小心把「未知」变成了「未找到」、把「隐藏」变成了「不存在」？
   - ReAct / agent 循环会不会误解这个返回结构？
   - 是否用测试/构建验证过，而不是只信任这次编辑？

7. **发现跑偏立刻沟通。** 如果发现原方向有缺陷，暂停并清楚说出来；不要因为已经开工就沿着弱方案继续写。

8. **收尾做一遍可靠性扫描：**
   - 扫描措辞不一致；
   - 扫描 compact/expanded、admin/user 等多状态之间的不一致；
   - 扫描本应在并行路径同时修的「单边修复」；
   - 跑一遍最小但有意义的验证。

## 升级（Escalation）规则

发现设计漏洞、语义冲突或不安全行为时，直接告诉用户：问题是什么、为什么重要、建议怎么改。在继续大范围编辑之前先做这件事。
